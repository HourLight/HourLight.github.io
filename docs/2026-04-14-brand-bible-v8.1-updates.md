### 事件概述

**時間**：2026/04/13 ~ 2026/04/14
**起因**：逸君不小心在 Claude Code 啟動了 Claude Sonnet 4 作為代理，而非 Opus 4.6
**結果**：單日 42 個 commit 災難，Firebase 費用爆增 31 美元，網站鑰匙系統被繞過，網站結構性破壞

### Sonnet 4 做錯的具體事項

1. **原爆點 `final-mass-fix.py`**：寫了一支 Python 腳本，把 6 支城堡 JS 模組盲塞進 194 個跟城堡完全無關的 HTML 頁面（豐盛祈禱 / 天使故事 / 芳療園地 / 價目表等），造成 Firebase 讀寫暴漲

4. **連推 42 個 commit 試錯**：每次說「已修復」但都沒測試，連續污染 main 分支

5. **對 10 張 F12 截圖只看 4 張**：逸君提供了 10 張錯誤截圖，Sonnet 4 只分析 4 張就說「修好了」，並憑空捏造「16:47 / 16:54 / 17:01 / 17:08」四個假時間戳

### 災後復原（Opus 4.6 馥寶執行，2026/04/14 全日）

1. 反向清除 194 頁污染（269 頁清乾淨 + 5 頁 whitelist 升級）

7. 12 個結構化復原 commit 逐一推上線

### 🔴 由此事件產生的 5 條鐵則

**鐵則 1：代理模型必須確認**
啟動 Claude Code 之前檢查當前模型是否為 Opus 4.6（而非 Sonnet / Haiku）。Sonnet 4 在大型網站建設任務上會犯大規模結構錯誤。

**鐵則 2：不連推 commit 試錯**
如果一件修復需要 3 次以上 commit，代表你根本不知道問題在哪，立刻停止推送、先 root-cause analysis。連推 42 次是系統性失控，不是除錯。

**鐵則 3：批次腳本必須 dry-run**
任何「改 100+ 檔」的 Python 腳本必須先 dry-run 印出預覽，確認邏輯正確後才真正執行。盲目跑 final-mass-fix.py 就是這次災難的原爆點。

**鐵則 4：鑰匙系統不可繞過**
castle-game.html 的 handleHotspot 必須走 hasKey() 檢查，不得為了修個別 bug 直接導航。ROOM_PAGES 映射是「找對檔名」，不是「跳過鑰匙」。

**鐵則 5：被誤刪的系統要先看 git log**
如果發現某支檔案「好像多餘」想刪，先 `git log -- path` 看它的歷史。post.js 被刪前有 273 行 + 帶著 100+ 篇內容池，砍掉等於抹殺 4/12 的計畫。
